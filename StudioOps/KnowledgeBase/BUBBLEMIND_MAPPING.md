# BubbleMind 原创系统映射

> 本文把中性系统总图映射到当前 Unity 工程。所有项目名称、角色、能力、数值、视觉和实现均为 BubbleMind 自有内容；参考证据只用于确认页面职责和系统分层。

## 1. 状态枚举

| 状态 | 含义 |
|---|---|
| `implemented` | 当前可运行垂直切片已实现并有既有验证 |
| `partial` | 有可运行子集，但缺少完整产品状态或正式权威服务 |
| `data_only` | 已有原创数据契约或档案展示，尚无完整玩法 |
| `spec_only` | 仅知识库或接口方向，未进入运行时 |
| `backend_required` | 正式实现前必须有后端或平台权威 |
| `deferred` | 不在当前模板近期范围 |

## 2. 当前原创架构

```text
ScriptableObject 内容
  -> GameDatabase / CharacterDefinition / CharacterContentProfile
  -> SkillDefinition / GachaBannerDefinition
       ↓
本地 Demo 服务
  -> GameStateService + JSON Save
  -> LocalGachaService
  -> LocalFormationService
       ↓
DemoGameController + 独立 Screen View
       ↓
BattleSimulation 产生有序 BattleEvent
       ↓
DemoBattlePresenter 重播原创角色、VFX、世界条和伤害数字
```

边界：本地服务是离线试玩替身。正式账号、经济、抽取、奖励、支付、活动、竞技和排行不能直接沿用它们。

## 4. 角色内容模板

一个 BubbleMind 原创角色由稳定身份键连接以下部分：

### 身份与呈现

- `characterId`
- 原创显示名、本地化键、称号、阵营和简介
- 稀有度、职业、元素、限定标记与内容审批状态
- 卡面、头像、模型预制体和展示资源引用
- 来源与原创性证明

### 战斗契约

- 基础属性、移动速度、攻击范围和目标偏好
- 基础攻击、三个运行时技能槽
- 被动、领域或同类扩展能力
- 状态标签、命中类型、目标规则和表现事件引用
- 怒气、冷却或其他资源由通用规则表达，不在 UI 硬编码

### 养成与获取

- 解锁条件和养成阶段
- 技能等级参数采用结构化记录，不把长篇文案作为规则引擎
- 原创获取来源、池关系和重复结果策略
- 玩家拥有、等级、阶段、解锁与装备保存在独立实例状态

### 资产交付

- 原创卡面、模型、材质、纹理、动作、VFX、SFX
- 通用动作接口和插槽
- 移动端三角面、材质、纹理、透明度和粒子预算
- 预制体生成、资源引用、技能连接和战斗出生验证

## 5. 跨页面单一数据链

```text
抽取 rewardId
  -> GameDatabase.GetCharacter(characterId)
  -> CharacterDefinition
       -> Portrait / Name / Rarity / Role
       -> Skill slots / Battle parameters
       -> CharacterContentProfile
            -> Archive / Growth / Acquisition / Extended abilities
  -> Character Archive / Gacha Reveal / Formation / Battle Request
```

规则：

1. UI 不保存另一份角色名称、卡面、稀有度或技能。
2. 抽取服务只返回身份和数量；卡面由目录解析。
3. 编成只保存角色身份键，不保存视图对象。
4. 战斗在开始时快照必要参数，表现层不反写模拟。
5. 补充档案不得重复角色主身份或成为战斗权威的第二来源。
6. 静态角色目录只读取 `characterId`；持有阵容以 `instanceId` 为列表身份并回查 `characterId`。
7. 角色详情分栏共享同一个选中角色上下文；切页不能复制或替换角色定义与玩家实例。
8. 玩法规则说明、编成校验、战斗上下文与结算资格必须引用同一个版本化模式 ID。

| 原创页面 | 已覆盖 | 下一步完整状态 |
|---|---|---|
| 首页 | 地图、资源、任务带和主要入口 | 主入口组与次级入口组、加载、功能锁定、通知聚合、当前位置和设置入口 |

## 7. 实施优先级

### 优先级 A：巩固原创内容模板

- 让每个角色档案都通过身份、能力、养成、获取和来源验证。
- 为技能时间线、目标规则和表现引用建立稳定数据契约。
- 为缺图、断链、重复 ID、未审批来源和非法枚举提供构建阻断。

### 优先级 B：补齐非经济产品状态

- 角色筛选、空态和长列表复用。
- 静态目录与持有阵容分离，并让详情分栏始终绑定同一个角色上下文。
- 编成预设、合法性错误和模式约束。
- 战斗暂停、速度、重播、性能和可访问设置。
- 扩展原创关卡定义与战斗上下文，使同一模式契约覆盖队伍规模、波次、跨波状态、终止和规则预览；不先引入正式奖励。

### 优先级 C：先建权威接口，再扩展经济

- 账号、库存、任务、活动、邮件、商店和抽取使用可替换接口。
- 所有请求带幂等键、内容版本和会话序列。
- 在后端安全模型、合规展示和失败恢复明确前，不把本地 Mock 扩成生产经济。
