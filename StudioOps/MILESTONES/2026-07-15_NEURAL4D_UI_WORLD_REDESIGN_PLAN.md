# Neural4D 与全局视觉重设计计划

> 日期：2026-07-15
> 状态：只读审计与测试计划完成；付费测试和实施等待授权
> 范围：Neural4D 候选管线、主页、角色页、关卡地图、战斗竞技场、共享 UI 视觉系统
> 当前权威项目计划：`../../PROJECT_PLAN.md`

## 1. 目标

把 BubbleMind 从当前的功能型 Demo 视觉，升级为“柔和动漫科幻 + 固定镜头 2.5D 战斗”的原创抽卡 RPG 体验，同时保留已经验证的抽卡、角色档案、五槽编队、怒气、技能和确定性战斗数据。

Neural4D 的建议定位是：

```text
角色与模块化场景件的高模/贴图候选供应商
        ↓
Blender 清理、分件、减面、形变、骨骼与 LOD
        ↓
Unity 页面舞台、Shader、VFX、Prefab、地图组装与性能验收
```

Neural4D 不负责按钮、字体、信息架构、完整关卡组装、技能 VFX 或最终 Unity 性能。

## 2. 已完成的只读测试

用户当前打开的 Neural4D Catherine 结果已进行无消费的旋转检查：

- 输入：一张包含正面、侧面、背面三只史莱姆的拼图。
- 输出统计：`550,756 vertices`、`1,000,000 faces`。
- Texture 与 PBR 已开启。
- 当前账户显示 `20 Power`，新生成按钮显示 `30 Power`。
- 正面主体的穹顶、凝胶裙边、黑洞位置和紫黑材质明显优于现有程序化 Blender 基线。
- 旋转后确认一大两小三只史莱姆会共同转动，说明拼图被雕成了三主体场景，而不是同一角色的真正多视图融合。
- 侧背面体积基本成立，但背部细节明显少于正面；轨道存在断裂、悬浮和穿插风险。
- 中央黑洞目前仍需确认是独立可维护结构，不能只凭正面发光贴图判定通过。

结论：Neural4D 值得进入受控试点，但当前结果只能作为视觉候选，不能直接进入 Unity 或用于批量角色生产。

## 3. 现有视觉层审计

### 3.1 可保留

- `DemoGameController` 已把主页、抽卡、角色、编队和战斗分离。
- `CharacterDefinition` 与 `CharacterContentProfile` 已能驱动卡面、属性、技能、档案和成长页。
- 1920x1080 CanvasScaler、安全区适配、基本转场和确定性战斗可继续使用。
- 当前 2D/3D 资产保留为回退，不在重设计阶段删除。

### 3.2 必须重做

- 主页目前是同一背景、标题、任务条和四个同权文字按钮，没有真正的 Hub、视觉焦点或入口层级。
- 所有非战斗页面复用同一张背景和深色遮罩，页面没有独立身份。
- 角色页固定为约 30% 双列卡库加 70% 文本详情，无法发挥实时 3D 模型，也把技能压缩成三条小文字卡。
- 当前战斗地图本质是 `480x270` 背景 Quad 加简单站位标记，不具备模块化、前中后景、环境动画或关卡扩展能力。
- UI 颜色、字号、锚点和英文文案大量硬编码；缺少图标、九宫格、真实字重和可复用主题层。
- 当前视觉以深蓝/紫为主，容易让地图、UI、角色和 VFX 混在同一色域。

## 4. 推荐产品方向

建议用一次可回退的 A/B 里程碑确认新方向：

### 路线 A：推荐

```text
二次元 3D 角色展示
+ 全幅 2D/2.5D Hub 与页面背景
+ 固定镜头 2.5D 战斗
+ 模块化 3D 场景前景与 VFX
```

优点：能利用 Neural4D 高模制作角色页、卡面和一致 Sprite，同时保留移动端性能与现有战斗结构。

### 路线 B：延期候选

完整实时 3D 角色与全 3D 地图。只有 Catherine 20K 低模、十单位战斗和第一张模块化地图在目标设备达到 60 FPS 后才进入评估。

当前 `PROJECT_PLAN.md` 仍把 Pixel2D 设为默认表现。用户确认路线 A 后，必须先更新主计划的视觉决策，再开始页面代码重做；在此之前不继续给旧像素 UI 叠加装饰。

## 5. 页面重设计

### 5.1 主页 Hub

- 全幅星渊观测庭场景作为第一视觉信号，不把主体验放进卡片。
- 左上：玩家头像、等级和名称。
- 右上：三种资源与设置入口。
- 左侧：当前章节、主任务和可领取状态。
- 场景热点：招募台、角色馆、战斗门；只显示真实可用入口。
- 底部：主页、招募、角色、编队、地图五项图标导航。
- 右下：独立高权重“挑战”命令，显示当前关卡。
- Demo Reset 移入设置菜单，不继续暴露在主页。

### 5.2 角色页

- 约 20% 角色列表、45% 实时模型舞台、35% 信息区。
- 中间舞台支持 Idle、拖动旋转、缩放、技能预览和卡面回退。
- 右侧标签：概览、技能、觉醒、档案。
- 技能采用图标列表加单项详情，不同时展示三段长文。
- 状态覆盖：已拥有、未拥有、满级测试、最长技能文本、模型加载失败。
- Catherine 使用独立黑洞/轨道/VFX 层，不能把全部表现烘进静态模型。

### 5.3 关卡地图

- 独立的浮岛/星轨章节地图，而不是直接从主页进入战斗。
- 节点状态：锁定、当前、已通关、可领奖、Boss。
- 页面包含章节标题、推荐战力、敌方预览、奖励和挑战入口。
- 地图概念图先作为视觉蓝图；正式场景由可复用模块件组装。

### 5.4 战斗竞技场

- 保留固定侧视/斜俯镜头和 20 格战场语义。
- 场景拆为玩法地面、站位层、边缘模块、远景天空、环境 VFX。
- Neural4D 只生成平台边缘、柱体、传送门、晶体、碎星等单体模块。
- 完整地图不得作为一个 AI 网格直接进入 Unity。
- 十单位开场、大招拉拽、击飞和后排瞬移必须保持清晰可读。

### 5.5 抽卡与编队

- 抽卡：全幅限定主题图、单抽/十连、保底/概率入口、2D 卡牌揭示。
- 编队：五个固定槽位、角色候选库、职业/元素筛选和战力反馈。
- 两页在主页、角色页和地图完成后统一迁移到新主题组件。

## 6. 视觉系统

主题：柔和动漫科幻观测庭。

| 用途 | 色值 |
|---|---|
| 珍珠白背景/高光 | `#F5F7FB` |
| 墨色正文/深层 | `#18202A` |
| 青色交互/水 | `#4CCBD1` |
| 珊瑚主命令/火 | `#FF6E62` |
| 金色稀有/奖励 | `#F2C85E` |
| 绿色成功/自然 | `#72C48B` |

- 紫色只用于虚空与 UR，不作为全局底色。
- 卡片与面板圆角控制在 `4-8px`，边界 1px。
- 稀有度通过边框、徽记和局部光效表达，不整卡灌色。
- 建立原创图标图集、九宫格边框、真实 Medium/Bold 字重。
- 页面转场 180-240ms，角色切换 120-180ms；提供跳过和减少动态效果。
- 触控目标不小于 48dp；所有数字区域预留稳定宽度。

## 7. Neural4D 正式测试矩阵

### Gate 0：法律与账户

在上传更多未公开设计前，必须获得或确认：

1. 付费 Studio/API 输入与输出完全排除训练。
2. 平台不得未经同意放入 Showcase 或按 CC BY 4.0 公开。
3. 删除、备份、API 文件的保留周期。
4. 商用权在退订后持续。
5. 当前权威 API 版本、端点、计费和并发。

公开 Terms 与帮助页目前存在训练、永久许可、主动公开和模板占位符冲突。Gate 0 未通过时，只能使用非敏感校准资产。

### Gate 1：单角色几何

- 输入：只含一只 Catherine 的干净正面图。
- 设置：High、1M faces、Quantity 1、先关闭 Texture/PBR。
- 连续三次，至少 2/3 只生成一个主体。
- 黑洞、角、凝胶裙边和轨道可读；无背景、基座、复制体。

### Gate 2：真正 MultiView

- Front、Back、Left、Right 分槽；不使用拼图，不镜像不对称视图。
- 至少 2/3 结果侧背面评分达到 4/5。
- 不出现接缝、重复角色、视图混合或多余基座。

### Gate 3：Texture/PBR

- 从最佳白模分支生成普通 Texture 与 PBR。
- 验证 BaseColor、Normal、Roughness、Metallic 和 UV。
- 切换至少三种灯光，确认贴图未烘焙固定高光和阴影。

### Gate 4：Retopology

| 资产 | LOD0 | LOD1 |
|---|---:|---:|
| Catherine 主体 + 黑洞/轨道 | <=20K | <=8K |
| 基础史莱姆 | <=2.5K | <=1K |
| 首张竞技场可见模块合计 | <=25K | 按距离裁剪 |

- 面数误差不超过 2%。
- 非流形边、零面积面和反法线为 0。
- Catherine 20K 轮廓与高模对比保持 95% 以上。
- AI Retopo 仅作候选；史莱姆形变区仍需 Blender 人工整理。

### Gate 5：分件、Refine 与导出

- 目标分件：Body、Horns、Core、Upper Orbit、Lower Orbit、Droplets。
- Refine 测试只修改一个局部，未指定结构漂移不超过 2%。
- 导出 GLB、FBX、BLEND 交叉检查；Unity/Blender 比例差异低于 0.1%。
- Pivot 归一到底部中心，材质不超过 6，贴图全部本地化。

### Gate 6：Unity

- Catherine 四个形变仍由 Blender 制作：`IdleBreath`、`Squash`、`Stretch`、`UltimateCollapse`。
- Character Page 转台在 1920x1080、1280x720 和 2400x1080 无遮挡。
- 十单位战斗在目标设备达到 60 FPS，关闭 VFX 后轮廓仍清晰。
- 模型失败时自动回退卡面，不阻塞角色页。

### Gate 7：地图与批量

- 使用基础水史莱姆验证族系复用。
- 使用一个平台边缘、一个柱体和一个传送门验证模块化地图。
- 角色与场景件测试均达到 80/100，才允许讨论批量上传。

## 8. 统一评分与硬失败

| 项目 | 分值 |
|---|---:|
| 身份与轮廓 | 25 |
| 360 度一致性 | 20 |
| 分件与可编辑性 | 15 |
| 拓扑与性能预算 | 15 |
| 贴图/PBR | 10 |
| Unity 导入与运行 | 10 |
| 稳定率与成本 | 5 |

通过线：80/100。

以下任一项直接失败：

- 出现重复角色或背景主体。
- Catherine 黑洞变成普通凹洞或纯贴图。
- 无法满足 LOD 预算。
- 主体、黑洞和轨道无法维护。
- 输入权利、保密或商业许可未澄清。

## 9. 实施阶段

### Phase 0：方向锁定

- 用户确认路线 A 或路线 B。
- 更新 `PROJECT_PLAN.md` 与 `CURRENT_MILESTONE.md`。
- 冻结旧 Pixel2D 视觉扩展，但保留回退资产。

### Phase 1：视觉稿先行

- 生成三张 1920x1080 原创视觉稿：主页、角色页、战斗竞技场。
- 每页最多两种有意义的方向，不做无边界变体。
- 用户选定后锁定色板、字体、布局和镜头。

### Phase 2：Neural4D 受控试点

- Catherine 干净单图与真正 MultiView。
- 基础水史莱姆。
- 一个地图平台模块。
- 完成高模、贴图、重拓扑、导出、Blender 和 Unity 审计。

### Phase 3：共享 UI 基础

- `UiTheme` 数据资产。
- 图标、字体层级、九宫格、按钮、标签、资源条、角色缩略卡和状态组件。
- 继续使用 uGUI，不在本轮迁移 UI Toolkit。

### Phase 4：主页与导航

- 建立全幅 Hub、资源条、任务、场景热点、底部导航和挑战 CTA。
- 完成安全区、键鼠、触控和回退图验证。

### Phase 5：角色页

- 建立 RenderTexture 模型舞台。
- 完成概览、技能、觉醒、档案以及拥有/未拥有状态。
- 接入 Catherine 候选或卡面回退。

### Phase 6：关卡地图与战斗竞技场

- 建立章节/节点数据与五种节点状态。
- 用模块件组装第一张竞技场；保留确定性 20 格战斗。
- 完成环境动画、前中后景、站位层和 VFX 可读性。

### Phase 7：抽卡、编队与完整回归

- 抽卡、编队、战斗 HUD、结算迁移到新主题。
- 完成 Generate、EditMode、PlayMode、Windows Build 和截图回归。
