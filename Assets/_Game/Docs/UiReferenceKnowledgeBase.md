## 清洁室边界

- 只记录信息层级、交互节奏、可读性和页面职责等通用 UX 原则。
- 不复制外部游戏的角色、立绘、场景、图标、字体、文案、布局坐标、数值、概率、代码、协议或资源命名。
- BubbleMind 的卡面由 GPT Image 按原创史莱姆设定生成；地图、角色数据与战斗规则继续来自本项目。
- 外部参考只回答“信息为什么易读”，不回答“怎样复刻这款游戏”。

## 观察结论

### 主界面

- 场景图本身就是导航语境，UI 应让背景保持可见，不用大面积不透明面板盖住地图。
- 主任务、常用系统与资源余额需要稳定分区；所有入口不应拥有相同视觉权重。
- 第一屏先回答“我在哪里、下一步做什么、当前资源多少”，说明性长文应移出主界面。

### 角色列表与详情

- 角色必须以可识别图像为第一信息，名称、稀有度、职业与拥有状态为第二层。
- 列表适合快速扫描，详情适合深读；二者应采用主从结构而不是把全部字段塞进同一张小卡。
- 详情页应保留一个明显的大角色视觉区，并把属性、技能与养成信息组织为稳定模块。
- 未拥有角色仍可查看档案，但必须清晰标记“未拥有”，不能伪装成已解锁。

### 抽卡

- 抽卡入口先展示池主题、目标角色、成本与概率入口；抽卡结果必须显示实际角色卡面，而非只有颜色或文字。
- RNG、扣费和重复角色处理归服务层；揭示动画与卡面只负责表现。
- 单抽结果需要同时说明角色、稀有度、职业、`NEW / DUPLICATE` 与剩余资源，避免玩家猜测发生了什么。

### 编队

- 五个槽位尺寸固定，并用卡面、职业和射程建立一眼可扫的队伍结构。
- 已选、可选、未拥有三种状态必须有不同的形状/明度反馈，不能只依赖一处小文字。
- 编队编辑与当前 3v5 测试夹具分离；UI 不应暗示五名保存角色都会在当前测试战斗中登场。

## BubbleMind 视觉系统

### 色彩令牌

| 用途 | 色值 | 规则 |
|---|---|---|
| 深色画布 | `#10131B` | 页面底色与高对比文字背景 |
| 玻璃表面 | `#171D28` / `#2B3442` | 低透明度分区，不遮蔽主要卡面 |
| 薄分隔线 | `rgba(221,232,244,0.16)` | 用边界建立层级，避免厚重金属框 |
| 主要强调 | `#61D5C1` | 选中、信息强调、可交互焦点 |
| 行动强调 | `#F06E61` | 进入战斗、确认抽卡等明确主命令 |
| 稀有强调 | `#F1C66D` | UR、概率与收藏价值；控制使用面积 |

### 排版与边框

- 正文和 UI 使用随包分发的 `Noto Sans CJK SC Regular`，不依赖玩家系统字体。
- 页面标题只用于真实页面层级；卡片内部采用更小、更紧凑的标题。
- 字距保持 `0`；不用全大写长句承担正文，不让文字压过卡面。
- 卡片圆角视觉控制在 8px 以内；当前 uGUI 以细线框、明度和留白建立分区。
- 主操作与次操作在颜色、尺寸和位置上分级，不做整排同权重按钮。

## 数据与页面契约

```text
Gacha rewardId
    ↓
GameDatabase.GetCharacter(characterId)
    ↓
CharacterDefinition
    ├─ Portrait / Name / Rarity / Role
    ├─ Stats / Description / Limited
    └─ Ultimate / Skill 2 / Skill 3
         ↓
Character Archive / Gacha Reveal / Formation / Battle Request
```

- `characterId` 是跨页面唯一身份键。
- `CharacterDefinition.Portrait` 是当前卡面唯一来源；同一角色在角色页、抽卡结果与编队中必须显示同一张图。
- 页面不能保存另一份概率、角色属性或拥有状态；它只从数据库与 `GameStateService` 读取。
- 每个角色数据必须拥有非空卡面和三项技能，验证器负责阻止缺图或断链内容进入试玩构建。

## 当前落地

- 首页：地图全幅背景、资源区、任务带与四个主导航入口。
- 角色档案：左侧双列角色卡库，右侧大卡面、身份、属性、拥有状态与三项技能。
- 抽卡：主题卡面、明确概率、成本、剩余水晶与带揭示动画的 2D 结果卡。
- 编队：五个稳定卡面槽位、职业/射程信息与可滚动候选库。
- 七张卡面和字体来源、许可证、尺寸与 SHA-256 均登记在 `ThirdPartyInventory.md`。
